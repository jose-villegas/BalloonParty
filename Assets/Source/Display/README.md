# Display

Camera and screen-adaptation logic.

## Contents

| File | What it provides |
|---|---|
| `OrthogonalSizeCameraController` | MonoBehaviour — sets the camera's orthographic size based on the device's aspect ratio. Reads `GameDisplayConfiguration` (injected) which maps aspect ratios to optimal orthographic sizes. Has a serialized `Camera` reference to avoid `Camera.main` lookup issues during additive scene loading. When `_continuous` is enabled, updates every `LateUpdate` to react to resolution or orientation changes; otherwise applies once on `Start` |
| `CameraShakeService` | MonoBehaviour — punches the gameplay camera on every `SpawnBlockedMessage` (a rejected balloon). The shake is an **additive offset** applied as a per-frame delta in `LateUpdate`, so it composes with whatever else drives the camera (the heart-drain pan just absorbs it) instead of fighting over the absolute position. Runs unscaled so slow-mo can't stretch it; skipped only while the current cinematic declares `CinematicTraits.BlocksShake` (the level-up states). Serialized `Camera` reference plus inspector-tunable duration / strength / vibrato. Registered via `RegisterComponentInHierarchy<CameraShakeService>()`; place it on the camera GameObject |
| `SceneCaptureService` | MonoBehaviour on the main camera — a **shared low-res capture of the scene's visuals**, bound globally as `_SceneCaptureTex` (resolution divisor from `IGameDisplayConfiguration`). Any effect that wants "what the screen roughly looks like" acquires it instead of using a `GrabPass` (a full-screen mid-frame resolve on tile GPUs); the first consumer is the Unbreakable's chrome reflection (`UnbreakableBalloonVariant` ref-counts via `Acquire`/`Release` on enable/disable). Renders only while someone holds an acquire, on a **time-based cadence**: `LateUpdate` accumulates `Time.unscaledDeltaTime` and fires once that accumulator passes `SceneCaptureFrameInterval / 60` seconds — the field is authored as "every Nth frame" but reinterpreted as seconds at a 60 fps reference, so the capture (and the screen-space GI chain it feeds) costs the same on a 120 Hz panel as it does on 60 Hz instead of doubling. Skipped frames keep the previous capture; the accumulator runs unscaled so cadence keeps advancing through pause/slow-mo. The serialized layer mask picks what gets captured — consumers sampling it should exclude their own layer to avoid feedback. The capture camera clears with **alpha 0**, so the RT's alpha channel doubles as a sprite-coverage mask (`ScreenSpaceLightService`'s occlusion source). The capture RT is allocated with a **24-bit depth buffer** even though nothing samples it — URP's RenderGraph rejects a depthless camera output texture ("Fake or uninitialized surface"). The runtime-created capture camera gets a `UniversalAdditionalCameraData` with post-processing off. One shared mask/resolution on purpose; a second consumer with different needs is the moment to generalize. Its inspector live-previews the capture RT in play mode (`SceneCaptureServiceEditor`). Registered via `RegisterComponentInHierarchy<SceneCaptureService>()`; **must be placed on the main camera in the Game scene** (scope build fails if absent) |
| `CameraBackgroundTint` | `[ExecuteAlways]` MonoBehaviour on the main camera — the scene light's diffuse term for the camera's solid clear colour (the `Sprite/Diffuse` response applied to the background). Owns the AUTHORED base colour so the multiply can never compound (`camera.backgroundColor` becomes derived state; `Reset()` adopts the camera's current colour, so adding it is visually a no-op). `Light Influence` eases unlit↔fully lit. Applies in `Update`, so the same-frame `LateUpdate` consumers of `backgroundColor` — the GI's ambient reference and the scene capture's clear (re-matched per capture) — read the tinted value. Reads colour/intensity from `ISceneLightSettings` (injected by VContainer); in edit mode (no DI) falls back to neutral white |
| `ScreenSpaceLightService` | MonoBehaviour on the main camera, next to `SceneCaptureService` (architecture: @ref arch_screen_space_light). Fakes a 2D global + multi-directional indirect light via three low-res blits: Pass 0 is an RSM-style 4-direction VPL gather (shadow = single toward-light march with mip penumbra; bounce = 4 cone marches at 90° spacing around the field's local direction, `_SecondaryWeight`-modulated). Pass 1 = 3×3 box soften, Pass 2 = temporal EMA. Composites over the frame as a camera-fitted multiplicative quad. All GI tuning lives on `SceneLightFieldSettings` SO (injected as `IScreenSpaceLightSettings` + `ISceneLightSettings`). DI-registered via `RegisterComponentInHierarchy` |

## How it works

`GameDisplayConfiguration` (in `Configuration/`) holds a reference world width and height. `GetOrthogonalSize()` computes the orthographic size that fits both against the current screen aspect (`max` of fit-width and fit-height), so the play field is fully visible on every device without letterboxing or clipping. Its custom editor previews the resulting size for common device ratios, in the inspector and as a Scene-view overlay.

The controller is registered via `RegisterComponentInHierarchy<OrthogonalSizeCameraController>()` in `GameLifetimeScope` and `LaunchLifetimeScope`. Place it on the camera GameObject in each scene with the camera reference wired in the Inspector.

## URP RenderGraph constraints

The project runs URP's 2D Renderer. Two constraints apply to anyone touching `SceneCaptureService` or `ScreenSpaceLightService`:

- **No mid-render-loop `Graphics.Blit`.** `RenderPipelineManager.beginCameraRendering` looked like the natural port target for the old `OnPreRender`-driven blit chain, but URP's RenderGraph rejects a `Graphics.Blit` issued from inside that callback ("EndRenderPass: Not inside a Renderpass"). `ScreenSpaceLightService` instead drives its three blits from `LateUpdate`, the same pattern the disturbance field already used — outside the render loop entirely, at the cost of reading the previous frame's capture (see above; it's invisible here).
- **Camera output render textures need a depth buffer.** A camera's `targetTexture` must carry a depth-stencil attachment even if nothing reads depth, or RenderGraph rejects it as a "Fake or uninitialized surface". `SceneCaptureService`'s capture RT is allocated with 24-bit depth for this reason alone — the 2D content underneath never samples it.

Both were discovered porting `ScreenSpaceLightService`; treat them as defaults for any new camera-to-texture or per-frame blit code in this pipeline, not special cases.
